## Находки

### 🔴 Критично — закрыть до передачи кому-либо

**Любой зарегистрированный пользователь может изменять и удалять ВСЁ в БД.**
На 8 таблицах (products, ingredients, product_ingredients, ingredient_groups, oils_meta, patented_formulas, formula_ingredients, subgroups) единственная политика `owner all <table>` = `ALL USING(true) WITH CHECK(true)` для роли `authenticated`. Раньше это было «ок» (единственный пользователь — Симона), но теперь регистрация самообслуживание (email-OTP) — любой человек может создать аккаунт и получить полный доступ на запись.

**Исправление:** [security_rls_lockdown.sql](security_rls_lockdown.sql) — чтение всем вошедшим, запись только uid владельца. **Запустить в SQL Editor до передачи ссылки.**

### 🟡 Важно

- **5 views с SECURITY DEFINER** (v_product_composition и др.) — обходят RLS читателя. Приложение их не использует; скрипт переводит в `security_invoker`.
- **`rls_auto_enable()` вызывается anon-ролью** через REST RPC — скрипт отзывает EXECUTE.
- **Leaked password protection выключена** — включить: Dashboard → Authentication → Settings → Password protection (HaveIBeenPwned). — [ ] Симона
- **UI:** «Режим редактора», «+ Средство», «+ Ингредиент», настройки (ключ remove.bg) были видны любому вошедшему. Теперь видны только владельцу (`isAdmin()` по email). Это косметика — реальная защита на RLS.

### 🟢 Норма / решения

- **Anon-ключ Supabase в коде** — это штатно для Supabase (он публичный по дизайну), безопасен ровно настолько, насколько строги RLS-политики. После запуска скрипта — ок.
- **Секретов в репо нет**: .env не закоммичен (есть только .env.example), ключ remove.bg вводится в UI и не хранится в коде, CloudPayments public id — публичный по природе.
- **10 таблиц «RLS без политик»** (brands, skin_types и др.) — закрыты полностью, приложение их не трогает. Ок.
- **Бакет product-images** позволяет листинг файлов — не критично (там только публичные картинки продуктов), можно закрыть позже.

## Видимость репозитория и имени

Факт: репо **public**, GitHub Pages включён, URL содержит имя (`simonakhatipova.github.io/Cosmetic_db`).

Варианты (по возрастанию усилий):
1. **Оставить как есть.** Секретов в коде нет; виден код и имя. Для теста с друзьями — приемлемо.
2. **Кастомный домен на GitHub Pages** (например, beautyhelper.ru / .app): имя исчезает из URL, репо остаётся публичным (код виден тем, кто целенаправленно ищет). Стоимость — только домен.
3. **Приватный репо + бесплатный хостинг** (Cloudflare Pages / Netlify деплоят из приватного репо бесплатно, GitHub Pages из приватного требует платный план). Код и имя скрыты полностью; деплой-процесс придётся перенастроить (вместо `npm run deploy`).

Рекомендация: для передачи первым тестировщикам достаточно №1 + RLS-скрипт; к запуску монетизации — №3 с доменом.

## Тестовая запись

После запуска RLS-скрипта тестовый пользователь сможет только читать каталог.

Создать: Supabase Dashboard → Authentication → Users → **Add user** → «Create new user»:
- email: например `beta@beautyhelper.test` (или реальная почта тестировщика)
- пароль: сгенерировать длинный
- ✓ Auto Confirm User

Вход — кнопка «Войти» на лендинге (почта+пароль). Либо тестировщик регистрируется сам через «Попробовать бесплатно» (нужен настроенный `{{ .Token }}` в шаблоне письма).
