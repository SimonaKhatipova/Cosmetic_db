# Личный кабинет + лимиты тарифов (freemium)

**Дата:** 2026-06-13
**Статус:** в работе
**Контекст:** приоритет монетизации из CLAUDE.md (freemium: база бесплатно, схема ухода — платно). Этот план — фундамент: где хранится тариф, как считаются недельные лимиты, экран профиля и мягкий paywall. Платная фича «подбор схемы ухода» — отдельным планом поверх этого.

## Лимиты (из тарифов на лендинге)

| | Бесплатный | Подписка (pro) |
|---|---|---|
| Поиск по базе | 7 / неделя | безлимит |
| Анализ своих средств (добавление состава) | 3 / неделя | безлимит |
| Сравнение | до 2 | до 5 + аналоги |
| Полный тест, подбор схемы, совместимость, дневник, рутина для кудряшек | — | да |

## Архитектура

**Тариф — server-authoritative.** Хранить в `user_metadata` нельзя: оно правится самим пользователем через `PUT /auth/v1/user`, и любой бесплатник сделал бы себя pro. Поэтому отдельная таблица `profiles` (RLS: читать только своё; UPDATE тарифа недоступен `authenticated` — меняет только service_role из платёжного вебхука или владелец вручную).

**Учёт использования** — таблица `usage_events` (kind = search|analysis, created_at). Недельный счётчик = COUNT за последние 7 дней. На MVP лимиты «мягкие»: проверяются и пишутся с клиента (worst case — продвинутый пользователь обойдёт логирование). Жёсткое серверное принуждение (RPC SECURITY DEFINER «проверь лимит и залогируй атомарно» или edge-функция) — позже, когда появятся платящие.

DDL — в [plans/2026-06-13-freemium-schema.sql](2026-06-13-freemium-schema.sql), запускает Симона в SQL Editor.

## Фазы

- [x] **Фаза 0 — схема БД.** `profiles` (id→auth.users, tariff, pro_until), триггер авто-создания профиля + бэкофилл существующих, `usage_events` + индекс, RLS. Применено и проверено (RLS на месте, 1 юзер → 1 профиль, триггер активен).
- [x] **Фаза 1 — загрузка профиля.** `loadProfile()` тянет `profiles`, глобаль `CURRENT_PROFILE`, хелпер `isPro()` (админ-владелец, либо tariff='pro' с непросроченным pro_until). Безопасная деградация: нет строки/таблицы → free.
- [x] **Фаза 2 — счётчики использования.** `loadUsage()` (COUNT поиск/анализ за 7 дней), `logUsage(kind)` (pro/владельца не пишет), локальный `usage` + `bumpUsage`. Поиск считается с дебаунсом 700мс по устоявшемуся уникальному запросу.
- [x] **Фаза 3 — гейтинг + paywall.** Компонент `Paywall` (бабочка, преимущества, «Оформить подписку»). Поиск сверх лимита → paywall + очистка строки. Сравнение сверх лимита — paywall (путь добавления из карточки ингредиента) / блок кнопки (карточка средства).
- [x] **Фаза 4 — экран «Личный кабинет».** `ProfileScreen`: тариф, индикаторы «X/лимит» (поиск, анализ), кнопка подписки (→ `PurchaseModal`), данные аккаунта, документы, выход. Вход — кнопка-аватар с пилюлей тарифа в шапке.
- [x] **Фаза 5 — сравнение 2 vs 5.** `compareLimit()` = 2 (free) / 5 (pro); прокинут в карточку как `compareMax`, корзина ограничена.

## Что осталось на будущее

- **Точка входа «анализ своих средств» для обычного пользователя.** Сейчас добавление продукта (анализ состава) доступно только владельцу в режиме редактора, поэтому счётчик `analysis` у бесплатника отображается (0/3), но триггерить его нечем. Когда появится пользовательский «анализ моего средства» — вызвать `overLimit('analysis')` перед действием и `bumpUsage('analysis')` после.
- **Жёсткое серверное принуждение лимитов** (RPC SECURITY DEFINER / edge-функция) — когда появятся платящие.
- **Платная фича «подбор схемы ухода»** — отдельным планом, гейтится `isPro()`.
