Чистка данных Supabase после миграции v4 (12 июня 2026). Скрипты в `plans/migration_v4/`: `008_data_cleanup.sql` (выполнен — ретиноиды, вода, дубли продуктов, группы), `009_alias_backfill.sql` (выполнен — опечаточные дубли + 645 строк по алиасам), `010_normalize_backfill.sql` (выполнен — нормализация, 430 строк закрыто, subgroups синхронизирован: 237 строк, 0 расхождений), `011_dedupe_final.sql` (выполнен — 328→193 unmatched), `012_final_polish.sql` (выполнен — 193→132 unmatched, явные близнецы, generic-слияние коллизий, U+2060), `013_ocr_spaces.sql` (агрессивный ключ только [a-z0-9/] — закрывает OCR-разрывы посреди слов, ~49 строк, остаток ~83 — настоящий мусор).

**Статус на 12.06.2026:** чистка данных завершена (с 1660 unmatched до ~83). Остаток — единичный мусор парсинга («чеснока», «краситель оранжевый», греческие/обрезанные строки), привязке не подлежит. Дальше — НЕ грузить новые продукты до починки матчера, затем доработка приложения.

**Ограничение Supabase SQL Editor:** не переваривает `CREATE FUNCTION ... AS $$...$$` (наивный сплит по `;`) — в скриптах только plain statements и TEMP TABLE, нормализацию инлайнить.

**Урок 010:** перед созданием «отсутствующего» ингредиента всегда искать по нормализованному ключу (без дефисов/регистра), а не точному имени — в 010 создались дубли к бездефисным двойникам, пришлось сливать в 011.

**Важно про концентрации:** если в `raw_inci_name` состава есть концентрация (ppm/%/IU) — она значима, строку не удалять и raw не менять; привязывать только `ingredient_id` к чистому каноническому ингредиенту.

**Корневая причина дублей — УСТРАНЕНА 12.06.2026:** матчер INCI в `App.jsx` переписан (план `plans/2026-06-12-inci-matcher-v2.md`): exact → нормализованный ключ → агрессивный OCR-ключ, по `inci_name` + `aliases`, пишет `match_source` (`exact`/`norm`/`ocr`). Заодно починена загрузка справочника (раньше грузился только на вкладке «Ингредиенты» — добавление продукта с других вкладок шло с пустым справочником, всё уходило в unmatched). Импорт можно возобновлять после ручной проверки Симоной (тестовый продукт → проверить `ingredient_id`/`match_source` в БД).

**Конвенции данных:** canonical-ингредиент один, варианты написания — в `aliases` через `|`. Группы задаются в `ingredient_groups` по справочнику `subgroups` (Retinol → Витамины/Жирорастворимые формы/Витамин A (Ретиноиды), Bakuchiol → .../Бакучиол). При удалении ингредиента сначала чистить `ingredient_groups`.

**Осталось на потом (не блокеры):** ~83 unmatched-мусора; «слипшиеся» ингредиенты (два вещества в одной записи) на ручную проверку: 5061, 5062, 5070, 5071, 5073, 5078, 5079, 5081, 5083, 5086, 5089, 3881, 4046, 4523, 4733; разрезанный по запятой «2-Oleamido-1,3-Octadecanediol»; косметика canonical-имён (регистр, «Benzophenone - 4»). Новые ингредиенты из 010/012 имеют `is_verified=false` — проверить группы. `ingredients_backup_pre_v3` (5252 строки) — снапшот перед v3, можно `DROP` после стабилизации.

**Проверенная формула нормализации** (для переноса в матчер App.jsx): lower → translate кириллических гомоглифов `авекмнорстух`→`abekmhopctyx` → убрать `_-*` → убрать не-ASCII `[^ -~]` → убрать скобки `\([^)]*\)` → срезать хвост-концентрацию `\d+...(ppm|ppb|%|iu)$` → схлопнуть слэши и пробелы. Агрессивный вариант (для OCR) — оставить только `[a-z0-9/]`, матчить лишь при length>=4 и единственном кандидате.
